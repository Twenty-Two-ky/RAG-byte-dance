## 1. 项目概览

### 1.1 项目背景

这是一个来自**字节跳动训练营的结题项目**,针对软件测试领域的痛点:

- 📄 **文档理解耗时**: PRD文档冗长,人工提取测试点效率低
- 🕳️ **用例覆盖不全**: 容易遗漏边界值、异常场景
- ♻️ **经验难以复用**: 历史优质用例无法自动关联
- 😵‍💫 **AI生成幻觉**: 通用大模型不理解业务规范

### 1.2 核心价值定位

构建 **"精准检索 → 智能共创 → 对抗评估"** 的完整闭环:

```
┌──────────────┐      ┌──────────────┐      ┌──────────────┐
│  RAG精准检索  │ ───→ │  多模态生成  │ ───→ │  AI质检评估  │
│  (去噪过滤)   │      │  (Human-in-  │      │  (Critic对抗)│
│              │      │   the-Loop)  │      │              │
└──────────────┘      └──────────────┘      └──────────────┘
        ↑                                           │
        └───────────── 反馈优化循环 ─────────────────┘
```

### 1.3 技术栈总览

| 层级 | 技术选型 | 说明 |
|------|---------|------|
| **前端框架** | Streamlit 1.51.0 | Python原生Web框架,快速构建数据应用 |
| **后端语言** | Python 3.10 | AI领域主流语言 |
| **LLM服务** | Google Gemini API | 支持1M+ Token上下文,多模态能力 |
| **向量数据库** | ChromaDB 1.3.5 | 轻量级本地向量库,保障数据隐私 |
| **数据处理** | Pandas 2.3.3 | 数据清洗与格式转换 |
| **容器化** | Docker + Docker Compose | 简化部署,环境隔离 |

#### 2.3.2 Prompt工程模式 (prompts.py)

**策略**: 统一管理所有Prompt模板,实现**意图路由**

```python
class PromptManager:
    CORE_SYSTEM_PROMPT = """
    【意图判断与响应策略】
    1. 场景A: 闲聊/通用问答 -> 自然对话,不输出JSON
    2. 场景B: 测试用例生成/修改 -> 严格JSON格式
    """

    @staticmethod
    def get_rag_filter_prompt(query, chunks):
        """LLM细筛: 过滤无关检索结果"""
        return f"""
        用户需求: {query}
        检索片段: {chunks}
        【要求】只保留技术相关内容,剔除噪音。
        """
```

**亮点**:
- **解耦业务与Prompt**: 修改策略时无需改动核心代码
- **意图路由**: 自动识别用户是闲聊还是工作指令

#### 2.3.3 RAG漏斗式检索架构 (核心创新!)

```python
# 第一步: 向量粗筛 (召回Top-K)
raw_results = rag_engine.search_context(query, n_results=3)

# 第二步: LLM细筛 (去噪过滤)
filter_prompt = PromptManager.get_rag_filter_prompt(query, raw_results)
filtered_text = llm_client.get_response(filter_prompt)

# 第三步: 注入生成Prompt
final_prompt = PromptManager.get_initial_prompt(prd_text, filtered_text)
```

**对比传统RAG的优势**:
| 传统RAG | 本项目的漏斗式RAG |
|---------|------------------|
| 直接召回 → 拼接Prompt | 召回 → **LLM去噪** → 拼接 |
| 容易包含无关片段(如百科、新闻) | 只保留业务相关的技术规范 |
| 浪费Token,干扰生成 | 纯净上下文,提升生成质量 |

## 3. 核心功能与代码实现

### 3.1 功能一: RAG知识库管理

#### 代码位置: `core/rag_engine.py`

**关键类**: `TextSplitter` (文本切片器)

```python
class TextSplitter:
    @staticmethod
    def recursive_split(text, chunk_size=500, chunk_overlap=100):
        """递归切片,支持滑窗重叠"""
        if len(text) <= chunk_size:
            return [text]  # 短文本直接返回

        chunks = []
        start = 0
        while start < len(text):
            end = start + chunk_size
            # 智能边界: 在句号/换行符处切分
            if end < len(text):
                separators = ["\n", "。", "!", "?"]
                for sep in separators:
                    if sep in text[end:end+50]:
                        end = text.index(sep, end) + 1
                        break
            chunks.append(text[start:end])
            start += (len(chunks[-1]) - chunk_overlap)
        return chunks
```

**亮点分析**:
1. **滑窗重叠**: 避免关键信息在切分边界丢失
2. **智能边界**: 在自然语义断点切分,保持上下文完整
3. **防御性编程**: 处理极端情况(文本过短/重叠过大)

### 3.3 功能三: AI对抗评估 (Critic模式)

#### 代码位置: `core/evaluator.py`

**设计思想**: 引入第二个AI作为"质检专家",形成生成者与评审者的对抗。

```python
class Evaluator:
    def evaluate_cases(self, prd_text, current_cases, rag_context, golden_cases):
        """
        评估维度:
        1. 覆盖度: 是否覆盖PRD所有功能点?
        2. 逻辑性: 步骤与预期结果是否自洽?
        3. 查重: 是否存在逻辑重复的用例?
        4. 规范性: 是否遵守RAG中的技术规范?
        5. 对比: 与Golden Sample的差距?
        """
        model = genai.GenerativeModel(
            model_name,
            system_instruction=PromptManager.EVALUATOR_SYSTEM_PROMPT
        )

        prompt = PromptManager.get_evaluation_prompt(
            prd_text, current_cases, rag_context, golden_cases
        )

        response = model.generate_content(prompt)
        report = extract_json_from_text(response.text)

        return {
            "score": 85,
            "summary": "整体质量不错,但缺少异常场景",
            "coverage_gap": ["未覆盖库存不足的场景"],
            "logic_issues": [{"id": "TC_003", "issue": "预期结果与步骤不符"}],
            "duplicates": ["TC_005与TC_009测试点重复"],
            "suggestions": ["建议增加SQL注入测试"]
        }
```

**价值分析**:
- **自动化质检**: 替代人工逐条验收
- **标准化度量**: 多维评分,可量化的质量指标
- **差异对比**: 若上传标准答案,自动计算偏差

#### 问题2: "遇到过什么技术难点?如何解决的?"

**难点1: ChromaDB的文本切片死循环**

**问题**:
> 初版`TextSplitter`在处理极短文本或极大重叠时会陷入死循环,导致程序卡死。

**分析**:
```python
# 问题代码
while start < text_len:
    chunk = text[start:start+chunk_size]
    chunks.append(chunk)
    start += (chunk_size - overlap)  # 如果overlap >= chunk_size,start永远不会增加!
```

**解决**:
```python
# 改进后
if len(text) <= chunk_size:
    return [text]  # 短文本直接返回,避免进入循环

step = len(chunk) - chunk_overlap
if step <= 0:  # 防御性检查
    step = 1
start += step
```

**收获**: 学会了防御性编程,处理边界条件。

**难点2: Streamlit的Session State管理**

**问题**:
> Streamlit每次交互都会重新运行整个脚本,如何保持对话历史?

**解决**:
```python
# 初始化
if 'gemini_history' not in st.session_state:
    st.session_state['gemini_history'] = []

# 每次对话后更新
response, new_history = llm_client.chat(user_input, history)
st.session_state['gemini_history'] = new_history
```

**收获**: 理解了Streamlit的无状态特性,学会使用`st.session_state`实现状态持久化。

## 7. 面试准备要点

### 7.1 RAG相关问题

#### Q1: "什么是RAG?为什么需要RAG?"

**标准答案**:
> RAG (Retrieval-Augmented Generation) 是检索增强生成。大模型的知识来源于训练数据,存在以下问题:
> 1. **知识截止**: 无法回答最新的信息
> 2. **领域知识缺失**: 不懂企业内部规范
> 3. **幻觉问题**: 容易编造不存在的内容
>
> RAG通过在生成前先检索外部知识库,将相关信息注入Prompt,从而提升准确性和时效性。

**进阶回答** (结合本项目):
> 在我们项目中,RAG的作用是让AI理解企业的技术规范(如密码策略、安全规则)和历史测试用例。我们还在传统RAG基础上增加了LLM过滤层,解决了检索噪音问题。

#### Q2: "向量数据库的原理是什么?"

**核心概念**:
```
文本 → Embedding模型 → 向量 (如768维浮点数组)
      ↓
  存储到ChromaDB (支持向量相似度搜索)
      ↓
  查询时: query向量 与 库中向量 计算余弦相似度 → 返回Top-K
```

**公式**:
```
余弦相似度 = (A · B) / (||A|| * ||B||)
```

**实际应用** (本项目):
```python
# 1. 向量化存储
embeddings = embedding_model.encode("技术规范文档")
chroma.add(documents=["..."], embeddings=[embeddings])

# 2. 检索
query_embedding = embedding_model.encode("用户需求")
results = chroma.query(query_embeddings=[query_embedding], n_results=3)
```

#### Q3: "如何评估RAG系统的效果?"

**指标体系**:
1. **召回率 (Recall)**: 检索到的相关文档 / 所有相关文档
2. **精确率 (Precision)**: 检索到的相关文档 / 检索到的所有文档
3. **生成质量**: BLEU/ROUGE分数(与标准答案对比)

**本项目的评估方法**:
- 上传标准用例(Golden Sample)
- Evaluator自动计算偏差
- 人工标注"是否遗漏关键测试点"

### 7.2 Prompt工程问题

#### Q4: "什么是Prompt Engineering?有哪些技巧?"

**定义**:
> Prompt工程是设计和优化输入文本,引导大模型生成符合预期的输出。

**常见技巧**:
1. **Few-shot Learning**: 提供几个示例
   ```
   示例1: 输入"登录功能" → 输出[{"id": "TC_001", ...}]
   示例2: ...
   现在请生成: "支付功能"
   ```

2. **思维链 (Chain-of-Thought)**: 让模型"说出思考过程"
   ```
   请一步步分析:
   1. PRD提到了哪些功能点?
   2. 每个功能点需要哪些测试场景?
   3. 生成测试用例JSON
   ```

3. **角色扮演**: 给模型设定身份
   ```
   你是一个拥有10年经验的资深测试架构师...
   ```

4. **格式约束**: 强制输出JSON
   ```
   【输出格式要求】必须是纯合法的JSON数组,不要包含Markdown代码块。
   ```

**本项目的实践**:
- 统一管理Prompt模板(`PromptManager`)
- 意图路由(区分闲聊与工作指令)
- 解释与数据分离(左侧思维链,右侧JSON)

### 7.3 Python相关问题

#### Q5: "Python的装饰器是什么?有什么用?"

**定义**:
> 装饰器是一种语法糖,用于在不修改原函数的情况下,增加额外功能。

**本项目实例**:
```python

    @st.cache_data: Streamlit的缓存装饰器

    ttl=3600: 缓存有效期3600秒
```

**手写装饰器**:
```python
def retry(max_attempts=3):
    """重试装饰器"""
    def decorator(func):
        def wrapper(*args, **kwargs):
            for i in range(max_attempts):
                try:
                    return func(*args, **kwargs)
                except Exception as e:
                    if i == max_attempts - 1:
                        raise
                    print(f"重试 {i+1}/{max_attempts}")
        return wrapper
    return decorator

@retry(max_attempts=3)
def call_gemini_api():
    # 可能失败的API调用
    pass
```

#### Q6: "解释Python的生成器(Generator)"

**定义**:
> 生成器是一种惰性计算的迭代器,使用`yield`关键字。

**优势**: 节省内存(不需要一次性加载全部数据)

**本项目潜在应用**:
```python
# 如果要处理超大文件,可以改造为生成器
def read_large_file(file_path):
    """逐行读取,而非一次性加载到内存"""
    with open(file_path, 'r') as f:
        for line in f:
            yield line.strip()

# 使用
for line in read_large_file("large_prd.txt"):
    process(line)
```
